## High Priority Issues

### H-1. `Plot.html` — Stored XSS via `taskEcho` / `feedbackMessages` rendered through `innerHTML`

**[system-google-sheets-addon/Plot.html: lines 4802–4855, 5403–5405]** — Server-reflected user strings rendered unsanitized

```
severity  : HIGH
check     : SEC-020
owasp     : A03 — Injection
file      : system-google-sheets-addon/Plot.html:4802
message   : innerHTML assignments render server response fields (stmt, ctx) that originate from attacker-controlled taskEcho.task without escaping.
evidence  : stmtEl.innerHTML = gStmt; ctxEl.innerHTML = gCtx; (×6 instances across lines 4802-4855)
fix       : Route all dynamic values through escHtml() before innerHTML assignment, or switch to textContent for plain-text fields.
```

- **Root cause**: The `stmt` and `ctx` variables are built from server response data that includes `taskEcho.task` — a string the user controls with no server-side sanitization. The `gStmt`, `gCtx`, `stmt`, `ctx` variables at lines 4802–4855 and 5403–5405 are assigned directly to `innerHTML` across 10 distinct call sites.
- **Impact**: An attacker who sets `task.task` to `<img src=x onerror=google.script.run.deleteSheet()>` achieves script execution in the GAS sidebar context. In GAS, XSS can invoke any server function exposed via `google.script.run`, executing with the user's full Google OAuth permissions.
- **Exact fix**:
```javascript
// Before (Plot.html:4802)
stmtEl.innerHTML = gStmt;

// After
stmtEl.innerHTML = escHtml(gStmt);
// OR for plain text:
stmtEl.textContent = gStmt;
```
Apply to all 10 instances at lines: 4802, 4803, 4804, 4853, 4854, 4855, 5403, 5404, 5405, 5435.

## Architecture & Design Observations

1. **Divergent sanitization maturity between WordPress and GAS clients**: The WordPress `app.js` consistently uses `escHtml()` for every user-derived string before `innerHTML` — a clean, auditable pattern. `Plot.html` in the GAS addon has **zero** `escHtml()` calls across 20+ `innerHTML` assignments. This suggests the GAS sidebar was built without the same security review standards. The two codebases need unified coding standards.

2. **Server-side trust boundary is missing**: `main.gs` treats all task fields as trusted once they arrive. There is no input validation layer between the client call (`google.script.run.pmcEstimatorAPI(payload)`) and the processing functions. A dedicated `validateTask(task)` function at the entry point would centralize type coercion, length limits, and character filtering.

3. **No Content Security Policy in GAS sidebar**: Google Apps Script HTML service runs in a sandboxed iframe, but the lack of any meta CSP tag in `Plot.html` means inline scripts and `innerHTML` injection face no browser-level mitigation. While GAS imposes `IFRAME` sandbox mode, a `<meta http-equiv="Content-Security-Policy">` with `script-src 'self'` would add defense-in-depth against injected `<script>` tags.

4. **No output encoding library in GAS codebase**: The WordPress plugin has `escHtml()` available globally. The GAS `Plot.html` file has no equivalent utility. Adding one is a prerequisite for fixing H-1 through H-4.

5. **Python agents are local-only CLI tools** — the subprocess, path traversal, and API key findings are lower risk than they would be in a web service context, but still worth hardening for defense-in-depth and CI pipeline safety.
